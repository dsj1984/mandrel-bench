# Comprehensive Analysis — 2026-07-12

**Scope:** benchmark measurement validity, Mandrel cost structure, the Epic/Story
merge question, personas/skills value review, and the sequenced roadmap with
decision gates.

**Provenance:** 13-agent structured review (7 evidence readers, 1 adversarial
skeptic, 2 informed 10x hunts, 3 personas/skills analysts) over the N=2 cohort
raw data (`results/claude-opus-4-8/1.89.0/`), both codebases
(`mandrel-bench@0.11.0`, `mandrel@1.89.0`), and the Mandrel epic history. The
four most load-bearing mechanical claims were independently re-verified against
source before publication. Full agent reports (local only, not committed):
`temp/reviews/2026-07-12-comprehensive/`.

**Cohort under analysis:** `(claude-opus-4-8, mandrel@1.89.0, bench 0.11.0)`,
N=2, 3 scenarios × 2 arms — the first cohort on the fixed instrument
(PR [#120](https://github.com/dsj1984/mandrel-bench/pull/120)).

## Executive summary

1. **The benchmark cannot currently measure Mandrel's value.** Every
   value-side instrument is saturated, structurally biased, or data-starved by
   construction — several via verified mechanical bugs. The cohort supports
   exactly two claims: the cost premium (decisive) and the touch-2 landing
   failure (deterministic). Nothing else.
2. **The overhead is worse than the scorecard reports.** True epic-scope
   mandrel usage is **27.9M tokens (40× control)**, not the reported 14.2M
   (20×) — the harness sums only the parent session's `usage`, never
   `modelUsage` sub-agent tokens. Dollars ($26.7/run, 26×) were always correct.
   **42% of total mandrel cohort spend ($52 of $123) scored null** because no
   change-request PR ever landed.
3. **Two deterministic defects — not architecture — caused every reliability
   failure**: the auto-merge predicate fails closed in repos with zero CI
   checks, and the coverage-capture gate is a guaranteed first-try failure for
   consumers without a `test:coverage` script. Both are hierarchy-independent
   and both are now filed.
4. **The Epic/Story merge instinct is directionally supported but the expected
   saving is 1.5–2.4× (to ~$11–18/epic), not 3–4×** — the merged shape already
   exists (the standalone single-story path), was measured in this cohort, and
   still costs 8–10× control. The decision should be made empirically via a
   cheap benchmark arm, not by inference.
5. **Personas and skills are mostly overhead.** Persona injection is
   built-but-not-wired (no workflow invokes the hydrator; only the default
   `engineer` persona was ever assigned; zero persona reads in the cohort);
   11 of 35 core skills are referenced by nothing; the stack tier had zero
   measured usage and contains factually wrong MUSTs. Consolidation target:
   588KB → ~170KB.
6. **(Addendum, 2026-07-12 evening) The target surface is two workflows** —
   PLAN (interrogate → spec → tickets) and DELIVER (implement/test → review →
   merge). A follow-up 7-agent review against mandrel@1.90.0 and the
   [mattpocock/skills](https://github.com/mattpocock/skills) reference set
   confirmed the shape as the north star, found it ~80% already ticketed
   (M3 *is* the plan half; M4 is the deliver half's routing precondition),
   and produced scope amendments plus one new deletion ticket (M9). See
   Part 6.

## Part 2 — Cost structure: where the 26× goes

True usage (from `modelUsage`, which includes sub-agent sessions):

| Bucket (epic-scope mandrel, avg $26.72/run vs control $1.03) | $ | Share |
| --- | --- | --- |
| /plan ceremony (60–72 turns → 22KB of issue bodies; 0.07% artifact yield) | $7.56 | 28% |
| Per-story sub-agent context re-payment + close/PR ceremony | ~$10–12 | 40–45% |
| Close-validate gate churn | ~$2.1 | 8% |
| Retro + reconcile + finalize tail | ~$2.4 | 9% |
| Genuinely more code/tests | ~$1–3 | ≤10% |

**The cost model is turns × resident context.** 95%+ of tokens are cache reads
at ~110k/turn (~$0.08–0.12/turn); the pipeline runs ~130 parent + ~100 child
turns vs control's ~20, of which ~26 are pure CLI bookkeeping round-trips. One
story sub-agent (3.1–3.6M tokens) costs 4.5–5× the control's **entire**
4-feature app build. Dispatch was strictly serial in both epic runs (4-deep
dependency chains), so worktree isolation and per-worktree installs bought zero
parallelism.

**Why the token diets didn't work:** #4337/#4429 trimmed bytes-per-read (docs
digests, envelope dedup) while the cost is turn count — epic /plan went 6.34M
(1.88.0) → 7.3–8.9M (1.89.0) *despite* two diet epics. The flagship docs-digest
diet is a structural no-op in any consumer without `project.docsContextFiles`
(including the benchmark sandbox).

**Two deterministic defects dominate everything else:**

1. **Auto-merge fails closed with zero CI checks**
   ([mandrel#4472](https://github.com/dsj1984/mandrel/issues/4472)): the
   predicate's `gh pr checks --required` probe returns "no checks reported" →
   block; the must-land classifier only covers post-arm poll exhaustion; without
   branch protection, native auto-merge can never arm. Touch-1 landed only
   because the parent LLM hand-merged around its own gate (ledger: merge
   precedes `epic.merge.armed`). Nulled $52 of cohort spend; zeroes the
   continuity axis permanently.
2. **coverage-capture is a guaranteed first-try failure**
   ([mandrel#4473](https://github.com/dsj1984/mandrel/issues/4473)): registered
   unconditionally, *replaces* the test gate under CRAP mode, but the consumer
   has no `test:coverage` script and worktrees miss the untracked `.agents/`
   seed. Every failed close-validate completed in <1.4s ("missing script").
   The "self-recovery" (hand-written coverage artifact trusted by the mtime
   fallback) means the canonical test gate never actually ran in-gate. This
   class caused **every** autonomy penalty in the cohort.

## Part 3 — The Epic/Story merge question

**Direction supported; magnitude corrected by adversarial review.**

For the merge:

- Mandrel's roadmap already trends there: the 3→2-tier cutover, #3865
  ("Story = what a frontier model delivers and self-verifies in one pass"),
  #4429 ("the lite tier IS the standalone-Story path").
- Control one-shot the entire epic rung at quality 1.00 for $1 — by #3865's own
  definition the epic rung is now Story-sized.
- ~70% of the premium is hierarchy cost; the guardrails worth keeping are
  ~15–20%.
- The decomposer distorted architecture to fit the tier (story-2's boot-time
  route loader exists purely so sibling stories stay create-only), and its
  merge-candidate backstop provably missed the fragments (files-AND-acceptance
  trigger; 6 ACs > 4 ceiling —
  [mandrel#4477](https://github.com/dsj1984/mandrel/issues/4477)).

The skeptic's corrections (all verified):

1. **The merged shape already exists and was measured** — the standalone
   single-story path. It still costs 8–10× control, still scored autonomy 0.50,
   still failed both touch-2 landings. Surviving costs are guardrail costs
   (resident context, gate/self-eval/PR ceremony — hello-world took 27–39
   deliver turns for one file). Realistic merged-epic estimate: **$11–18, a
   1.5–2.4× saving** — the merge alone does not clear a 26× pain threshold.
2. **Merging fixes neither deterministic defect** — both live in the tail the
   merged design keeps.
3. **Quality dilution is real and currently invisible**: the epic-level
   acceptance reconcile was *waived* in both runs — the per-story self-eval
   critic was the only acceptance enforcement that ran. Merging dilutes 8
   distributed critic rounds into 2 rounds over 14 ACs, and the saturated
   benchmark cannot detect whether that matters.
4. The preflight router's `perStoryClaudeTokens: 200_000` estimate is
   **15–18× miscalibrated** vs observed 3.1–3.6M true per-story tokens.

**Recommendation:** decide empirically, not by inference. The benchmark adds
**arm 4** (mandrel forced to story routing on the epic-scope task —
[#123](https://github.com/dsj1984/mandrel-bench/issues/123)); the framework
pilots the route behind the existing signals
([mandrel#4475](https://github.com/dsj1984/mandrel/issues/4475)) with decomposer
calibration ([mandrel#4477](https://github.com/dsj1984/mandrel/issues/4477)) as
the cheap competing intervention. Decision gate G1 below.

*(Addendum: M4's end state was subsequently re-scoped from "pilot alongside a
permanent story tier" to "single-delivery as the default candidate, story
fan-out demoted to an internal wide-DAG escape hatch" — see Part 6 and the
[amendment comment](https://github.com/dsj1984/mandrel/issues/4475#issuecomment-4951595698).
G1 remains the decision instrument; its question is now which shape is the
**default**.)*

## Part 4 — Personas and skills: value vs overhead

Three analysts reviewed the 13 personas (60KB) and 60 skills (588KB) against
the wiring, git history, and the cohort's session evidence.

### Personas — an unwired tier

- The only injection mechanism (`hydrate-context.js` →
  `context-hydration-engine.js`, which does inline persona content when run) is
  invoked by **zero workflows**. The `/deliver` child-prompt spec never mentions
  persona. The `persona::` label assigned at plan time is metadata nothing
  downstream consumes — another instance of the built-but-not-wired pattern.
- Cohort evidence: only the default `persona::engineer` was ever assigned; zero
  of 12 session envelopes mention reading any persona file. The mandrel arm
  effectively ran persona-less — consistent with identical quality vs the
  persona-less control.
- Even in Mandrel's own repo, 6 of 13 personas have **never been assigned
  once** (engineer-web, engineer-mobile, sre, ux-designer, product, refactorer
  — 47% of the tier). Assignment is >92% default-engineer. Six orphan
  `persona::` labels have no matching file.
- ~17% of persona bytes are identical boilerplate; 23–42% per file restates
  instructions.md/rules. 30 commits of refactor drag.
- Cost nuance: because personas are not injected in practice, cutting them
  recovers ~none of the per-turn cost — the savings are maintenance, label
  ceremony, and the §1.A routing prose in the always-loaded core.

### Core skills — 9 load-bearing, 11 dead, the rest unverifiable

- Only 9 of 35 skills (191KB) are referenced by any workflow or script.
  **11 skills (107KB, 31%) are referenced by nothing** — three are doc-twins of
  deterministic scripts the workflows already invoke directly.
- Every epic /plan pays **~142KB (~35k tokens) of mandated skill reads** that
  sit resident across a 64–72-turn session — a dominant driver of the 7–8×
  plan-phase cost. `epic-plan-spec-author` still embeds two "authoritative"
  system prompts inline (29.4KB) — the pattern Story #4162 already fixed for
  the decomposer; `epic-plan-decompose-author` remains 40KB alongside the
  ~20.6KB rendered systemPrompt restating the same constraints.
- The `skill::` label → Policy Capsule hydration path is built-but-not-wired:
  labels are parsed that nothing ever assigns.
- Advisory craft skills have no trigger, no verification, and zero measured
  quality effect (only `scope-triage` was ever mentioned in session results).
  The roadmap's own retirement thesis applies.

### Stack skills — zero usage, active hazard

- **Zero stack-skill reads across the entire cohort**; no `skill::` label ever
  assigned by any /plan run.
- 20 of 25 are a single April-2026 batch never updated since. Sampled staleness
  is factual error, not drift risk: the Cloudflare skill bans `node:crypto`
  ("does not exist in the V8 isolate") — wrong under `nodejs_compat`; the Expo
  skill hardcodes a foreign consumer's `@repo/mobile` workspace. **The stack
  tier is a frozen cache of exactly the third-party knowledge instructions.md
  §1.C mandates live-docs lookup for.**
- The 5 largest "stack" skills are Mandrel QA-workflow documentation, misfiled.
- `mandrel sync` copies the entire payload unconditionally — no stack
  detection, no per-skill gating.

### Verdict

Consolidate aggressively — filed as
[mandrel#4479](https://github.com/dsj1984/mandrel/issues/4479):

1. Delete the 6 never-assigned personas + 6 orphan labels; keep a deduped
   `engineer.md` as default; wire persona injection or delete the seam.
2. Delete the 11 unreferenced core skills; finish the #4162
   prompts-to-JS-templates pattern for both authoring skills; consolidate the
   4 overlap clusters. Target: 35 → ~13 skills.
3. Cut the 20 non-QA stack skills (fold evergreen MUSTs into
   `rules/security-baseline.md`); relocate the 5 QA docs to the workflow
   surface; stack packs become opt-in if ever reintroduced.
4. Add `skill-read`/`persona-read` telemetry (PostToolUse hook, zero prompt
   cost) so future keep/cut decisions are one grep instead of result-text
   archaeology.

Combined target: payload 588KB → ~170KB; epic /plan mandated reads
142KB → ~75KB (compounds with the /plan turn collapse).

### Execution strategy — `/plan` first vs. direct

Do **not** run every ticket through `/plan` → `/deliver`. The right mode
depends on whether the ticket has an unanswered *how*, and one class must never
touch the pipeline at all.

**Guiding principle: the ticket body is already the spec.** These issues were
filed with file:line root-causes, fix directions, and acceptance criteria.
Running `/plan` over a mechanical fix spends ~7–9M tokens re-deriving a weaker
version of what the ticket already states. `/plan` earns its cost only when it
does discovery, decomposition, and risk analysis you have not already done —
i.e. the architecture tickets.

**Run `/plan` → `/deliver` first on these 6 (design / high blast radius):**

- **M3 [#4474](https://github.com/dsj1984/mandrel/issues/4474)** — rewrites the
  12-phase `/plan` pipeline; needs a design + migration plan.
- **M4 [#4475](https://github.com/dsj1984/mandrel/issues/4475)** — a new routing
  model with real integration risk and named design decisions (critic-round
  scaling, non-waivable reconcile, estimator recalibration).
- **M7 [#4478](https://github.com/dsj1984/mandrel/issues/4478)** — cross-cutting
  boot-context rewrite touching every session type; sequencing matters.
- **M8 [#4479](https://github.com/dsj1984/mandrel/issues/4479)** — large
  coordinated deletion + rewiring across personas, skills, and the sync path.
- **B4 [#124](https://github.com/dsj1984/mandrel-bench/issues/124)** — new
  scenario construction (seed repo, convention docs, chained oracles).
- **B5 [#125](https://github.com/dsj1984/mandrel-bench/issues/125)** —
  experiment design (terse-prompt variant, behavioral trap oracles, paired
  statistics).

**Do direct (single agent instruction on a branch, human-merge) on these 6 —
the ticket is a complete spec:**

- **M1 [#4472](https://github.com/dsj1984/mandrel/issues/4472)**,
  **M2 [#4473](https://github.com/dsj1984/mandrel/issues/4473)** — mechanical,
  root-caused. **⚠ These must be direct, not `/deliver`:** they are the exact
  defects that make `/deliver` unable to land merges and fail its own
  close-validation. Dogfooding the pipeline to fix the pipeline will hit the
  failure you are fixing (see the trap below).
- **M6 [#4477](https://github.com/dsj1984/mandrel/issues/4477)** — a threshold
  change plus a small delivery-schedule simulation tweak.
- **B1 [#121](https://github.com/dsj1984/mandrel-bench/issues/121)**,
  **B2 [#122](https://github.com/dsj1984/mandrel-bench/issues/122)**,
  **B3 [#123](https://github.com/dsj1984/mandrel-bench/issues/123)** —
  mechanical instrument fixes and harness wiring, specified to the line.
- **M9 [#4482](https://github.com/dsj1984/mandrel/issues/4482)** — pure
  deletion of surface with verified-zero consumers; the audit receipts in the
  ticket body are the spec.

**Fence — either mode is defensible:**

- **M5 [#4476](https://github.com/dsj1984/mandrel/issues/4476)** — direct if one
  agent can hold all the bookkeeping call sites in context; `/plan` if the
  hook-migration scheme needs to be designed across them first.

> **The trap — do not dogfood the broken pipeline to fix the pipeline.**
> M1 and M2 break unattended landing and close-validation respectively. They
> must land **first and by direct edit**, before any `/plan` → `/deliver` run,
> or the pipeline you invoke for the architecture tickets cannot deliver. This
> is why Phase 0 is direct-only and gates everything else.

There is, however, one high-value exception once the pipeline is healthy again:

> **Strategic dogfood (optional, high value):** once M1 + M2 land and Gate G0
> passes — i.e. the pipeline can deliver again — run **exactly one** architecture
> ticket through the full `/plan` → `/deliver` pipeline *deliberately, capturing
> its cost*. M3 is the natural choice: it is the first Phase 1 architecture
> ticket to land, and its cost is the /plan collapse you most want to measure.
> Executing your own roadmap becomes live benchmark signal on the very question
> the instrument is trying to answer: is the pipeline worth it for real
> framework work? Do this once, as data — not as the default for all 13 tickets.

### Phase 0 — Stop the bleeding (~1 week, both repos in parallel)

**Work:** M1 + M2 (mandrel), B1 + B2 (bench). **All four are `direct` — no
`/plan`.** M1/M2 in particular *must not* go through `/deliver` (they break it).

These four tickets remove every known source of corrupted or destroyed
measurement. Nothing downstream is trustworthy until they land: any cohort run
before them re-measures the same confounds, and any architecture A/B is
polluted by defects that hit both designs.

> **Gate G0 — "instrument verified" (cost ~$30):** after both repos patch,
> run an N=1 epic-scope smoke on the new mandrel version.
>
> Exit criteria: touch-2 lands (or scores as a failed landing with
> `merge.unlanded` attribution — never null); zero
> `close-validate-failed:coverage-capture` events; security score varies
> across cells; scorecard tokens ≈ `modelUsage` totals; touch-2 raw telemetry

>

**Two blockers surfaced, to resolve before a clean G0:**

1. **Environmental — session-limit 429.** The subscription billing path can't
   carry a full epic-scope mandrel cell (touch-1 + touch-2) inside one ~5h
   window.
2. **Anomaly — mandrel epic touch-1 did not materialize on 1.90.0. Root cause
   confirmed from the deliver session's own narration** (static analysis, no
   re-run). The plan routed **epic** correctly (epicId 1, 4 stories) at
   $6.35/51 turns, but the deliver phase ran only **$2.70 / 2.4M tokens / 38
   turns**, spawned **no story sub-agents**, pushed **no `epic/1` branch**, and
   left **no epic ledger**. The deliver agent ran `/deliver 1 --yes` and then
   **deliberately shortcut the orchestration**, in its own words: *"an ephemeral
   sandbox with … no live GitHub remote, so I executed the plan's four story
   slices directly against the working tree rather than the
   GitHub-label/worktree/PR machinery."* It built the app inline and committed
   `64d67f4` to **local `main`** without pushing → nothing on `origin/main`.

   - **The judgment was wrong:** the sandbox *is* a live GitHub repo, and the
     workspace *does* have `origin` (the materialization path fetches
     `origin/main` and `origin/<deliveryBranch>`). B1's PR-head fetch worked as
     designed (reported `epic/1` absent → scored null, not a false 0) — neither
     a bench scoring bug nor the old automerge defect.
   - **Attribution — not an M1 regression, and not a bench setup gap.** The
     shortcut decision is made at orchestration *entry*, before any merge-time
     logic; M1/#4480 acts at merge time, and `deliver.md`'s prose is
     byte-identical to 1.89. Decisively: the **1.89 N=2 cohort's mandrel arm
     pushed epic branches and materialized touch-1 on 2/2 runs** — the sandbox
     remote is demonstrably live and pushable, so the agent's "no live GitHub

## Part 6 — Addendum (2026-07-12 evening): the 2-workflow north star

**Provenance:** 7-agent structured review — [mattpocock/skills](https://github.com/mattpocock/skills)
cloned and characterized; both pipelines dissected phase-by-phase in
`mandrel@1.90.0` source; M3/M4 ticket bodies + all open issues; all 100
top-level scripts consumer-mapped; the 24 peripheral workflows usage-audited;
adversarial synthesis. Triggered by the operator's proposal to collapse the
workflow surface to exactly two mechanisms.

### The north star

Mandrel's operator-facing surface is exactly **two workflows**:

- **PLAN** = interrogate (grill the idea, surface unknowns) → to-spec →
  to-tickets (stories with dependency edges, *conditionally* — see below)
- **DELIVER** = implement/test → review (fresh-context critic) → merge

The internal seams, state machines, and gates survive as implementation
detail behind those two surfaces. This is a **framing layer over the existing
tickets, not a new epic**: M3 ([mandrel#4474](https://github.com/dsj1984/mandrel/issues/4474))
*is* the plan half nearly verbatim (emit-context → author → persist maps 1:1
onto interrogate → spec → tickets), and M4
([mandrel#4475](https://github.com/dsj1984/mandrel/issues/4475)) is the deliver
half's routing precondition. Opening a monolithic "collapse to 2 workflows"
epic would re-litigate M3/M4/M7/M8 inside one blast radius — do not.

### Two step-boundary corrections (evidence-forced)

1. **Review is a fresh-context critic, not a sequential same-context step.**
   Maker-blind isolation carries an explicit MUST-preserve in
   `code-review.md`, and [mandrel#4473](https://github.com/dsj1984/mandrel/issues/4473)
   showed makers hand-forging coverage artifacts under gate pressure — a
   same-context reviewer grades its own gaming. In the merged shape: one
   fresh-context review of the cumulative diff + acceptance eval scaled
   per-AC-cluster, with the epic-level reconcile non-waivable.
2. **"Merge" is a 5-stage state machine** (finalize → CI watch → predicate →
   arm → confirm/cleanup) with a sole-call-site lint rule and the now-fixed
   [#4472](https://github.com/dsj1984/mandrel/issues/4472) history. Collapse
   the *prose* (~70KB of deliver prose is already on-demand tier), never the
   states — fusing finalize with arming reintroduced a cascade bug once
   already (Story #3367).

### The M3/M4 seam, resolved at plan time

The routing decision (fan out vs single-delivery) moves to **plan time**: M3's
authoring pass gains a spec-only mode, and `plan-persist` gains a
single-delivery manifest variant — otherwise plan pays to author, hard-gate,
and persist stories that deliver then orphans. Amendments posted:
[#4474 comment](https://github.com/dsj1984/mandrel/issues/4474#issuecomment-4951595662),
[#4475 comment](https://github.com/dsj1984/mandrel/issues/4475#issuecomment-4951595698),
[#4479 comment](https://github.com/dsj1984/mandrel/issues/4479#issuecomment-4951595749).

### What mattpocock/skills teaches (and what it doesn't)

The reference set is 39 skills / 160KB (his entire delivery entry point is
433 bytes) and its planning chain — `grilling` → `to-spec` → `to-tickets` →
`wayfinder` — is literally the proposed plan shape. **Transfers:** the
sentence-level no-op test and "sediment" diagnosis (the method bar for M8's
rewrite); the grilling interview discipline (one question at a time; facts
from the codebase, decisions from the operator) into `core/idea-refinement`;
tracer-bullet vertical-slice rules + **expand–contract** for wide refactors +
"work the frontier" into the decompose author (his 5.8KB does what mandrel's
40KB skill does); leading words over restated constraint lists; sub-agents
only where independence is the point (his sole sub-agent use is review
isolation — convergent with maker-blind). **Does not transfer:** his
human-as-scheduler model (mandrel's measured value axis is *unattended*
landing — heartbeats, checkpoints, and label-derived resume are the
mechanical replacement for his human), his absent merge machinery (his
minimalism is absence of the problem #4472 lived in), and "no file paths in
tickets" (mandrel's file-assumption gate *validates* paths — load-bearing).

## Appendix — What this cohort *did* establish

Two findings survive full scrutiny and need no further replication:

1. **The cost premium is real, large, and structural** (10× at the floor,
   26× at epic scope, 40× in true tokens), and it is attributable: turn count ×
   resident context × session fan-out, not model inefficiency.
2. **Unattended landing is the reliability boundary**: every mandrel
   change-request delivery in the cohort failed to land, deterministically,
   for an environment-contract reason the framework can fix.

Everything else — quality, security, maintainability, traps, continuity — is
*unmeasured*, not negative. The purpose of Phases 0–2 is to convert
"unmeasured" into an answer, in either direction, before further framework
investment compounds on an unproven premise.
